Architecture v2 · Customer-Hosted · Speech Routing

Virtual Receptionist
Everything in the Customer's Tenant

Caller speaks a name → Azure Speech Recognition transcribes it → Graph API fuzzy-matches against an Azure AD Security Group → Direct Routing transfers the call to that Teams user. Every resource lives in the customer's Azure subscription.

⚠ Deployment notes: ACS resources must be provisioned with --location global (not a regional value) — data residency is set separately via --data-location. The acs_callback_url in App Configuration must include ?code=YOUR-FUNCTION-KEY — without it, ACS mid-call events receive HTTP 401 and the bot cannot process speech or complete transfers. Teams notifications use Power Automate webhook URLs — Office 365 Connector URLs (*.webhook.office.com) are deprecated.

☎️ Call Flow — Step by Step
1
Caller dials the Teams resource account phone number (PSTN Direct Routing).
2
ACS receives the call and fires IncomingCall to /api/incoming_call?code=KEY.
3
Function answers call. ACS TTS plays greeting via SSML. Speech recognition starts.
4
ACS Speech Recognition listens. Caller speaks a name. RecognizeCompleted fires to /api/acs_callback?code=KEY.
5
Function queries Graph API: loads staff names + AAD Object IDs from AD Security Group (paginated, 5-min cache).
6
Three-stage match: Exact → Soundex phonetic → rapidfuzz fuzzy. Confidence margin check prevents false positives.
7
TTS confirms: "Connecting you to Sarah Jones." Transfer stored in _pending_transfers dict. PlayCompleted fires.
8
On PlayCompleted, transfer initiated via transferCallToParticipant(MicrosoftTeamsUserIdentifier). No race condition.
9
Call arrives in staff member's Teams client. Event logged to App Insights.

🔑
Permission GUIDs are resolved from Graph SP at deploy time — not hardcoded. Ensures correctness across tenants.
🏷️
Function App uses System-assigned Managed Identity for Key Vault and App Config — no credentials in code.
⚠ Scale-out limitation — _pending_transfers dict: The bot uses a module-level dict to track pending transfers between the "Connecting you to…" prompt and the actual transfer call. This is correct for a single Function App instance. If the Consumption plan scales out, a PlayCompleted callback may arrive on a different instance than the one that stored the transfer, silently dropping it. At typical receptionist call volumes (one concurrent call) this is not a practical problem. For high-volume deployments, replace _pending_transfers with Azure Table Storage keyed on callConnectionId.

👥 Azure AD Security Group — How Staff Routing Works

Group name
VirtualReceptionist-Staff
Security group in Entra ID. Customer IT admin manages membership.
Adding a new staff member
Add them to the group
Azure AD → Groups → VirtualReceptionist-Staff → Members → Add. No code changes, no redeployment. Live within 5 minutes.
Removing a leaver
Remove from group
Bot will no longer match their name on the next call. Instant effect after 5-minute cache expires.
What Graph returns
displayName + id
Bot loads displayName for fuzzy matching and Object ID for the transfer target. extensionAttribute1 for TTS pronunciation override.
Fuzzy match threshold
> 65% confidence
Below threshold → transfer to default reception. Configurable via App Configuration key receptionist:match_threshold.
Pagination
Supports >999 members
Graph returns up to 999 members per page. Bot follows @odata.nextLink for groups larger than 999 members.
